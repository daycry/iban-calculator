# Internationalization (i18n)

This resolves the open question from Â§13 of the design spec: *"Define message keys (`messageKey`) and
whether they are translated via CI4's `Language/`, given that the default `message` can be English in
the standalone core."*

## Decision: English core, optional CI4 translation layer

Every validation `Violation` carries **two** pieces of text:

```php
final readonly class Violation
{
    public function __construct(
        public ViolationCode $code,     // stable, backed enum â€” e.g. ViolationCode::ChecksumFailed
        public string $messageKey,      // stable string key â€” e.g. 'iban.violation.checksum_failed'
        public string $message,         // human-readable default text, hardcoded English
    ) {
    }
}
```

- **`code`** is the machine-stable identifier. Switch on this in application code â€” it never changes
  and is enum-typed, so PHPStan/IDEs catch typos.
- **`messageKey`** is a stable, translation-ready string key (`'iban.violation.<snake_case_code>'`),
  namespaced under `iban.violation.*`. It exists specifically so a consuming CI4 application can look
  it up in its own `Language/` files.
- **`message`** is the **default, English-language** human-readable text, hardcoded directly in
  `Daycry\Iban\Core\Validator`. This is deliberate: the core (`src/Core`, `src/DTO`, `src/Enums`, etc.)
  is framework-free and must work standalone â€” outside CodeIgniter 4, there is no `lang()` function and
  no `Language/` directory to fall back to. A core that returned `null` or an empty string for
  `$message` without CI4 would be a worse default for the majority of standalone users than a hardcoded
  English sentence.

Why not put the strings in a CI4 `Language/en/Iban.php` file and call `lang()` from the core? Because
`lang()` is a CodeIgniter global function â€” calling it from `src/Core/Validator.php` would violate the
unidirectional dependency rule enforced by `tests/Architecture/CoreIsFrameworkFreeTest.php` (see
`CLAUDE.md`). The core must not know CodeIgniter exists.

## The 8 message keys

All defined in `Daycry\Iban\Core\Validator`, one per `ViolationCode` case:

| `messageKey` | Default `message` (English) |
|---|---|
| `iban.violation.blank` | The IBAN is empty. |
| `iban.violation.too_short` | The IBAN is too short. |
| `iban.violation.illegal_characters` | The IBAN contains illegal characters. |
| `iban.violation.unknown_country` | Unknown or unsupported IBAN country code. |
| `iban.violation.bad_length` | The IBAN length is invalid for its country. |
| `iban.violation.malformed_structure` | The IBAN structure is malformed. |
| `iban.violation.checksum_failed` | The IBAN check digits are invalid. |
| `iban.violation.national_check_failed` | The national check digits are invalid. |

See [`docs/usage.md`](usage.md#validation-and-the-8-violationcode-cases) for when each one fires.

## Translating in a CI4 application

v1.0 does not ship a bundled `Language/` directory (there is nothing in `src/Language/` today) â€” the
`messageKey` is provided so a consuming application can wire up its own translations without waiting
for the package to do it. A typical approach:

```php
// app/Language/es/Iban.php
return [
    'violation' => [
        'blank'                  => 'El IBAN estĂˇ vacĂ­o.',
        'too_short'               => 'El IBAN es demasiado corto.',
        'illegal_characters'      => 'El IBAN contiene caracteres no vĂˇlidos.',
        'unknown_country'         => 'CĂłdigo de paĂ­s de IBAN desconocido o no soportado.',
        'bad_length'               => 'La longitud del IBAN no es vĂˇlida para su paĂ­s.',
        'malformed_structure'      => 'La estructura del IBAN estĂˇ mal formada.',
        'checksum_failed'          => 'Los dĂ­gitos de control del IBAN no son vĂˇlidos.',
        'national_check_failed'    => 'Los dĂ­gitos de control nacionales no son vĂˇlidos.',
    ],
];
```

```php
$result = service('iban')->validate($input);

if (!$result->isValid()) {
    $violation = $result->firstViolation();

    // Prefer the app's own translation, keyed by messageKey; fall back to
    // the core's English default if no translation exists for the active locale.
    $localized = lang($violation->messageKey);
    $text = $localized !== $violation->messageKey ? $localized : $violation->message;
}
```

(`lang()` returns the key itself unchanged when no translation is found for it, which is what the
fallback check above relies on â€” this is CI4's own documented `lang()` behavior, not something this
package adds.)

This keeps translation entirely opt-in and entirely at the CI4 layer: the framework-free core stays
translation-agnostic and dependency-free, while `messageKey` gives integrators a stable, English-free
identifier to build a full i18n experience on top of.

## Why not translate `ViolationCode` itself?

`ViolationCode` is a backed `string` enum whose `->value` (e.g. `'checksum_failed'`) is meant to be a
**stable API contract** â€” safe to serialize to JSON (see the `--json` output of `iban:validate`), log,
or compare against in application logic across releases. Translating it would break that contract.
`messageKey`/`message` exist precisely so display text can vary independently of the stable `code`.
